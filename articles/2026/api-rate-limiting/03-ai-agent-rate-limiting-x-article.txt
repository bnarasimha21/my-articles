I Got Locked Out of My Own GitHub. An AI Agent Did It.

I was using Claude Code to clean up my GitHub repos. Updating descriptions, batch-organizing things I had been putting off for months. Perfect job for an AI agent.

In minutes, the agent swept through dozens of repos. Reading metadata. Updating descriptions. Pushing changes. Done before my coffee got cold.

Then I asked it to list all my repositories.

Access denied.

GitHub's API gateway flagged my activity. Dozens of rapid-fire API calls in quick succession. I looked like an attacker. Legitimate, authenticated, paid-account usage. Indistinguishable from a DDoS attack.

Got access back after a short cooldown. No lasting damage.

The infrastructure protecting APIs was designed for humans. That world is ending.


Why Agents Break Rate Limiting

Traditional rate limiting. Fixed requests per minute. Applied uniformly. Excess gets throttled.

Works for humans. Developers make a few calls per page load. Mobile apps sync in small intervals. Steady and predictable.

AI agents do not work like that.

When an agent decides to clean up all repositories, it does not pace itself. It identifies every repo, evaluates each one, fires off API calls in rapid succession. Not malicious. Just efficient.

To an API gateway, efficient and attack look identical.

Here is what makes this harder. Attackers are using AI to look more human-like. Legitimate AI agents generate traffic that looks more bot-like. The signals are inverting.

A human makes 50 API calls in an hour.

An AI agent doing the same work makes 500 in five minutes.

Same total work. Completely different traffic pattern.


Three Actors, One Rate Limiter

Humans. Predictable, slow, session-based. What rate limiters were designed for.

Legitimate AI agents. Fast, bursty, authenticated. Traffic patterns that resemble attacks. Operating on behalf of real users.

Malicious bots. Using AI to mimic legitimate traffic. Some now mimic agent-like patterns to hide among legitimate agent traffic.

Traditional rate limiting treats this as binary. Within limits or not. The harder question is who is making these calls, why, and should we allow this specific burst.

The industry is starting to respond. Adaptive rate limiting that considers your history. Agent identity protocols. Intent-aware gateways. The direction is right. Most APIs have not caught up yet.


If You Are Building APIs

Create separate rate limit tiers for agent traffic. Higher burst limits, same total quota.

Implement transparent rate limit headers. X-RateLimit-Remaining and X-RateLimit-Reset. Agents can self-regulate if they know where they stand.

Design for burst, not just throughput. Sliding window beats fixed window. Token bucket is even better.

Distinguish read and write operations. An agent reading 100 repos is very different from writing to 100 repos.

Log agent actions separately. When something breaks, you need to know if a human or an agent caused it.


The Bigger Picture

We are moving from APIs serving humans through apps to APIs serving agents on behalf of humans. The infrastructure we built over 15 years was designed for the first world.

The second world needs different infrastructure. Not just higher limits. Fundamentally different approaches to identity, intent, and trust.

Imagine an AI agent managing a production deployment or orchestrating a financial transaction. Getting rate-limited mid-operation because the gateway could not tell it apart from an attack.

That is not an edge case anymore. That is Tuesday in 2026.

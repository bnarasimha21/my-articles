Nix and Jarvis: How I Built My Personal AI Around One Claude Subscription

The only difference between us and Tony Stark is that he had unlimited tokens.

Everyone who has watched Iron Man has at some point pictured what it would be like to have a Jarvis. An assistant that is always there, knows everything about you, takes actions on your behalf, talks to you like a person.

Today's models can absolutely do that. The thing in the way is not the intelligence. It is the meter ticking every time you use them.

I have been chipping away at this for over a year. What I ended up with is Nix, my personal assistant, running on one Claude subscription.


A short history of trying

I was one of the earliest users of OpenClaw. For a while I had it wired to my Claude subscription. The closest thing to Jarvis I had ever used.

Then Anthropic announced the subscription could not be used with third-party agent frameworks. I lost the setup overnight.

I switched to Claude Code's own channels plugin system. A lot of plumbing. But it worked.

So I built Nix.

Nix is my personal assistant. Voice-first on my Mac. Available on my phone over Telegram. Answers questions. Takes actions.

Web search. My email. My calendar. Scoped folders. My GitHub. Can create files. Edit files. Write code. Open pull requests.

I talk to it the way I would talk to a smart engineer who already knows what I am working on.

The whole thing runs on one Claude subscription.


How it is different

Voice-first conversation, not voice notes. I talk. It answers. We have a conversation.

Claude subscription, no per-token meter. No API calls. No third-party harness.

Same assistant on mobile and Mac. Same memory. Same identity. Same project state.

Runs on my own Mac. The files are mine. The runtime is mine. The credentials never leave.


The technical core

Three properties.

One. A long-running Claude Code session. The CLI takes a session ID and lets me resume the same conversation across calls. No re-paying the system prompt and history every turn.

Two. tmux to make the session resilient. The primary Nix session lives in a named tmux window. Terminal closes, SSH dies, screen sleeps. The session is unaffected.

Three. A LaunchAgent so it survives reboots. macOS reboot, sleep, network blip. The session comes back automatically. The assistant is genuinely always-on with my Mac on.


Two surfaces

Phone is a small bun server talking to the Telegram Bot API. Pipes messages into the tmux session. Pipes replies back out. A hundred lines of code.

Mac is a web dashboard with a floating mic orb. Wake word "Hey Jarvis". Browser does speech-to-text. Sends text to a local endpoint that runs a subprocess against the same engine.

The voice surface has its own persona, Jarvis. The engine underneath is the same Nix.


Why two names then

If they share everything, why two names at all. Why not both Jarvis, or both Nix.

The names reflect posture, not implementation.

Jarvis is the voice agent. The desk companion. Short questions, instant answers.

Nix is the operator. The one I delegate to. Write this article. Ship that code. Summarise this thread.

The shared engine and memory let them hand off invisibly. The names keep the modes distinct in my head.

A contract with myself about what kind of help I am about to ask for.


One brain, two surfaces

The deeper trick is not the subscription. It is that Nix and Jarvis read from the same memory files. Same identity. Same project state. Same logged conversations.

Ping Nix from Telegram on the train. Ask Jarvis at my desk an hour later. Jarvis already knows what we talked about. Not because the two are talking to each other. Because they are reading the same files.

One me written down on disk. Any surface picks up where the last left off.

The memory layer is Markdown files in a known folder plus a SQLite log. The model reads them on session start. New decisions get added. Old facts get removed. Over months it becomes a real picture of me, written by me and the assistant together.


What it costs

The flat Claude subscription is the only Claude-side cost.

A tiny OpenAI text-to-speech bill for the voice. A cheap droplet for other agents.

That is it. No per-token meter. No surprise bill at month-end.


What it is not

Not a magical unlimited setup. Claude Code has its own rate limits per tier. For one person with a phone and a laptop, the headroom is large.

Not a workaround for Claude's terms. The subscription is for personal use. Not reselling. Not proxying. Not public.


Why it matters

Most people assume running a personal AI 24/7 means paying through the nose for tokens. The default mental model pushes you to "only when I have a specific question."

A subscription-powered setup flips it. The assistant runs all the time. Watches my work. Holds context across sessions. Does not penalise me for using it more.

The leverage is in removing the token-counter from my head.


Where to start

You do not need every piece on day one.

Get Claude Code installed. Learn the non-interactive mode with session resume. Until "one session, many turns" feels natural.

Pick one interface you already live in. Wire it to the resumable session. You now have an always-on assistant.

Add memory. A folder of Markdown files the assistant reads on start. A logger that captures the conversation.

Voice and dashboards are nice-to-haves. The first two steps are the unlock.

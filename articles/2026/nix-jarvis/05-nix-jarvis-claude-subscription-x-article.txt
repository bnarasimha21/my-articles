Nix and Jarvis: How I Built My Personal AI Around One Claude Subscription

I wanted an AI assistant that is genuinely always-on.

Not a chat window I open. Something I can ping from anywhere that already knows what I am working on.

Wire it to the API and pay per token. That works for an hour. It does not work for a real always-on setup. The token bill becomes the bottleneck before the assistant becomes useful.

So I built mine on top of one Claude subscription. Two assistants, one flat monthly cost.


What I have

Nix lives on my phone. When I message my Telegram bot, Nix replies. It runs as a Claude Code session in tmux on my Mac, kept alive by a Launch Agent.

Jarvis lives on my Mac. A small web dashboard with a floating mic orb. I say "Hey Jarvis", it listens, answers in voice, shows the response in a chat panel.

Two interfaces. Same engine. Same memory files. Same identity. Different surfaces.


The trick

Most personal AI setups force a choice. Use the API and pay per million tokens. Or use the chat app and accept it is not callable from code.

There is a third path. Claude Code is a CLI. Its non-interactive mode runs a prompt and returns a result. The CLI authenticates against the Claude subscription.

That means I can call Claude from any script, any background process, any bot. The cost is flat.

Nix is a Claude Code session that never quits. Jarvis spawns a subprocess when it needs heavy lifting. Both routes ride the subscription.


How it splits

Nix is one long-running Claude Code session. Each new Telegram message lands in the same tmux pane with full context already loaded.

Jarvis is the opposite shape. Every voice turn spawns a fresh subprocess that runs the prompt, returns the answer, and exits. Session resumption means each turn does not re-pay the system prompt and history.

Both routes ride the same Claude subscription. No API key on the side. No per-token meter.

Voice feels snappy. Background work runs without a meter.


One brain, two surfaces

The deeper trick is not the subscription. It is that Nix and Jarvis read from the same memory files. Same identity. Same project state. Same logged conversations.

Ping Nix from Telegram on the train. Ask Jarvis at my desk an hour later. Jarvis already knows what we talked about. Not because the two are talking to each other. Because they are both reading the same files.

One me written down on disk. Any surface picks up where the last left off.


Why two names then

If they share everything, why two names at all. Why not both Jarvis, or both Nix.

The names reflect posture, not implementation.

Jarvis is the voice agent. The desk companion. Short questions, instant answers, conversational.

Nix is the operator. The one I delegate to. Write this article. Ship that code. Summarise this thread.

Different expectations of how I work with each one. The shared engine and memory let them hand off invisibly. The names keep the modes distinct in my head.

A contract with myself about what kind of help I am about to ask for.


What makes it work

Three pieces.

Persistence. A session has to stay alive between messages. tmux plus a Launch Agent.

Memory. SQLite chat log for short term. Markdown files the model reads on session start for long term.

A personal knowledge base. A tool that ingests my Telegram chats and GitHub repos every ten minutes into a vector database. A prompt hook injects the most relevant past context before the model sees the question.

End result. The assistant can answer "what did we decide about that project last week" without me hunting for the chat.


What it costs

The flat Claude subscription is the only Claude-side cost.

A tiny OpenAI text-to-speech bill for the voice. A cheap droplet for the other agents.

That is it. No per-token meter. No surprise bill at month-end.


What it is not

Not a magical unlimited setup. Claude Code has its own rate limits per tier. For one person with a phone and a laptop, the headroom is large. For a hundred concurrent tasks it would not be.

Not isolated from Claude's terms either. The subscription is for personal Claude Code usage. Not reselling. Not proxying. Not public. The distinction matters.


Why it matters

Most people assume running a personal AI 24/7 means paying through the nose for tokens. The default mental model pushes you to "only when I have a specific question."

A subscription-powered setup flips it. The assistant runs all the time. Watches my work. Holds context across sessions. Does not penalise me for using it more.

The leverage is in removing the token-counter from my head.


Where to start

You do not need both pieces on day one.

Get Claude Code installed. Run a few non-interactive prompts. The CLI is the gateway, not the chat UI.

Pick one interface you already live in. Wire it to a subscription-priced call. You now have an always-on assistant.

Add memory. A folder of Markdown files the assistant reads on start. A logger that captures the conversation.

Voice and dashboards are nice-to-haves. The first two steps are the unlock.

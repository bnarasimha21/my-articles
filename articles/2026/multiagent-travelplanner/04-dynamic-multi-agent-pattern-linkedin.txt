Let the Model Decide: From Fixed Pipelines to Dynamic Agent Orchestration

I was building a travel planner with multiple AI agents. The first version had four of them hardcoded: flights, hotels, restaurants, attractions. Every query ran through all four, every time.

Someone typed "What's the weather in Bali next week?" and the system searched for flights, found hotels, recommended restaurants, and listed tourist spots. For a weather question.

That is when I threw out the fixed pipeline.

—

🔀 What I Changed

The new version has no fixed set of agents. One orchestrator reads the query, decides what tasks are actually needed, and spins them up on the fly.

A weekend beach trip gets 4 tasks.
A complex international family vacation gets 10.
The system matches its effort to the problem.

The planning logic is a prompt that tells the model how to decompose queries into independent parallel tasks. Not a class hierarchy. Not an interface. A paragraph of English.

—

⚡ The Speedup

All tasks run at the same time using asyncio.gather.

With 6 agents each taking 2-4 seconds:
→ Sequential: 15-20 seconds
→ Parallel: ~4 seconds
→ 4-5x improvement under good conditions

Real world is lower. API rate limits, concurrency caps, and cost scaling (10 agents = 10x the tokens) bring that down.

What keeps cost in check: powerful model for decomposition and aggregation, fast cheap model for individual agents. The orchestrator thinks carefully. The agents just research and return data.

—

🏗️ Prompt Plans, Code Catches Failures

The decomposition prompt is the blueprint. Swap it and the same code becomes a research assistant, a competitive analysis tool, or a document review system.

But the prompt only handles the happy path. Reliability lives in code. Schema validation. Retries. Timeouts. Cost guardrails.

The prompt decides what to do. The code decides what to do when things go wrong. In production, the second part matters more.

—

⚖️ Breadth vs Depth

More agents cover more ground, but each one gets a single pass. No iteration. No "go deeper on this."

Fan-out trades depth for breadth and speed. Right trade for travel planning and competitive analysis. Wrong trade for medical diagnosis or financial modeling.

—

⚠️ The Fragile Part

Everything depends on one LLM call: the decomposition.

I have seen it create duplicate tasks for the same query. Skip visa research for international trips. Spawn 12 agents when 3 would do.

Production systems need:
→ A validation step before execution
→ A task count ceiling to bound cost
→ Partial failure handling
→ Quality tracking: coverage, redundancy, user satisfaction

There is no single "correct" decomposition. Track what you can and tune over time.

—

✅ Where It Fits

Works when query complexity varies widely, tasks are naturally independent, and users are waiting for responses.

Does not fit when tasks have dependencies, quality requires multi-round iteration, or the domain is narrow enough that the same steps always apply.

—

I started with four hardcoded agents because it was simple. It worked until it did not. The queries were too varied. Some needed two agents, some needed ten.

Dynamic decomposition solved that. Three phases: decompose, execute in parallel, aggregate. The model handles the hard part. The code handles everything else.

The question is not how many agents you need. It is who should be deciding that.
